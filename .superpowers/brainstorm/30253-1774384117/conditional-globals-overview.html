Conditional Inputs and Workflow Globals

Recommended production design: typed references, one workflow-level Globals node, multi-rule condition groups.

Recommended

Typed References, Not Freeform Strings

The conditional node adapts its left-hand operand from whatever is connected to it. Users pick from valid semantic references instead of typing raw paths.

Why

  • Low runtime overhead because references are resolved from schema, not inferred on every render.
  • Safer UX because only valid operands are offered.
  • Extensible to globals, shell outputs, LLM outputs, and future node types.

Tradeoff

  • Needs a small typed reference model in workflow data, not just plain text config.

4. Model Summary

A

Typed operand references

Condition rules store semantic references like shell.exit_code or globals.deploy_env, not display strings.

B

Exactly one Globals node

Canvas-level workflow state declaration with defaults and override policy.

C

Multi-rule condition groups

One conditional node can express all or any rule evaluation.
